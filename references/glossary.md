# 기술 용어 비유 사전

> 사용자 프로파일의 직업에 맞는 비유를 선택하여 용어 첫 출현 시 괄호 안에 삽입한다.

---

## 1. CLI (Command Line Interface)

### HR/인사 담당자
Excel 수식처럼, 텍스트로 컴퓨터에 명령하는 방법입니다.
마우스 클릭 대신 키보드로 "이 파일 열어" 같은 명령을 칩니다.

### 백엔드 개발자
GUI 대신 터미널에서 모든 걸 제어하는 인터페이스입니다.
`cd`, `ls`, `git` 같은 명령어를 사용합니다.

### 프론트엔드 개발자
브라우저 DevTools 콘솔처럼, 텍스트로 시스템을 제어하는 인터페이스입니다.
npm, webpack 등을 실행할 때 쓰는 그 환경이에요.

### 풀스택 개발자
터미널에서 텍스트 명령으로 시스템을 제어하는 인터페이스입니다.
ssh, docker, kubectl 등 서버 관리도 CLI로 합니다.

### UI/UX 디자이너
Figma 단축키의 강력 버전입니다.
텍스트 명령어로 모든 작업을 빠르게 처리할 수 있습니다.

### 제품 기획자/PM
Jira에서 필터 검색하는 것처럼, 텍스트로 컴퓨터에 명령하는 방법입니다.
마우스 클릭보다 빠르게 작업을 처리할 수 있습니다.

### 데이터 분석가
SQL 쿼리처럼, 텍스트 명령으로 컴퓨터를 제어하는 방법입니다.
`python script.py` 같은 명령을 실행하는 환경입니다.

### 마케터/콘텐츠 제작자
Google Ads에서 검색어를 입력하듯, 텍스트로 컴퓨터에 명령하는 방법입니다.
마우스 클릭 대신 키보드 명령으로 작업합니다.

### 일반 사무/관리
카카오톡 메시지 보내듯, 텍스트를 입력해서 컴퓨터에 일을 시키는 방법입니다.
"이 파일 복사해", "저 폴더 열어" 같은 명령을 칩니다.

### 연구원/학생
논문 검색엔진에 키워드 치는 것처럼, 텍스트 명령으로 컴퓨터를 제어하는 방법입니다.
R이나 Python 스크립트를 실행할 때 쓰는 그 환경입니다.

## 2. 터미널 (Terminal)

### HR/인사 담당자
컴퓨터와 대화하는 텍스트 창입니다.
Mac의 "터미널" 앱 또는 Windows의 "명령 프롬프트"를 말합니다.

### 백엔드 개발자
Shell(bash, zsh 등)을 실행하는 CLI 애플리케이션입니다.
SSH 접속, 서버 관리, 빌드 등 모든 작업의 허브입니다.

### 프론트엔드 개발자
VS Code 하단의 통합 터미널과 같은 것입니다.
`npm start`, `yarn build` 등을 실행하는 그 창이에요.

### 풀스택 개발자
iTerm2, Warp 같은 터미널 에뮬레이터입니다.
로컬 개발부터 서버 배포까지 모든 명령을 실행합니다.

### UI/UX 디자이너
코드를 직접 실행할 수 있는 검은 화면입니다.
개발자들이 맨날 쓰는 그 화면이에요. 여기서 Claude Code도 실행합니다.

### 제품 기획자/PM
컴퓨터에 텍스트로 명령을 내리는 프로그램입니다.
개발팀이 "터미널 열어서 확인해볼게요" 할 때 그 화면입니다.

### 데이터 분석가
Jupyter Notebook 대신 명령어를 직접 치는 실행 환경입니다.
pandas, matplotlib 스크립트도 여기서 실행할 수 있습니다.

### 마케터/콘텐츠 제작자
메모장처럼 텍스트를 입력하는 프로그램인데, 입력하면 컴퓨터가 바로 실행합니다.
영화에서 해커가 쓰는 검은 화면 같은 것이에요.

### 일반 사무/관리
컴퓨터에 문자 보내는 앱이라고 생각하면 됩니다.
"이것 해줘"라고 치면 컴퓨터가 바로 실행합니다.

### 연구원/학생
R Studio의 콘솔 창과 비슷합니다.
명령어를 입력하면 바로 결과가 나오는 실행 환경입니다.

## 3. git

### HR/인사 담당자
"보고서_최종_진짜최종_v3.docx" 대신, 모든 변경 이력을 자동 관리하는 도구입니다.
누가 언제 무엇을 바꿨는지 자동으로 기록합니다.

### 백엔드 개발자
분산 버전 관리 시스템(DVCS)입니다.
브랜치, 머지, 리베이스 등으로 코드 이력을 관리합니다.

### 프론트엔드 개발자
코드 변경 이력을 관리하는 버전 관리 시스템입니다.
GitHub/GitLab에서 PR 올릴 때 쓰는 그 도구예요.

### 풀스택 개발자
코드 버전 관리 + 협업 도구입니다.
feature branch, hotfix, release 등의 워크플로우를 지원합니다.

### UI/UX 디자이너
Figma의 "Version History"와 같은 개념입니다.
파일 변경 이력을 저장하고, 언제든 이전 버전으로 돌아갈 수 있습니다.

### 제품 기획자/PM
Google Docs의 "버전 기록"을 모든 파일에 적용한 것입니다.
팀원 누가 무엇을 바꿨는지 추적하고, 실수하면 되돌릴 수 있습니다.

### 데이터 분석가
분석 코드의 실험 노트입니다.
"이 모델 파라미터 바꾸기 전" 상태를 저장하고, 필요하면 되돌립니다.

### 마케터/콘텐츠 제작자
Google Drive의 "이전 버전 보기" 기능을 모든 파일에 적용한 것입니다.
"지난주 카피가 더 좋았는데" 할 때 바로 되돌릴 수 있습니다.

### 일반 사무/관리
파일의 "실행 취소"를 무한대로 할 수 있는 도구입니다.
어제 작업, 지난주 작업도 되돌릴 수 있습니다.

### 연구원/학생
논문 작성의 변경 이력 추적 도구입니다.
"3일 전 서론이 더 나았는데" 할 때 바로 그 버전을 꺼낼 수 있습니다.

## 4. commit

### HR/인사 담당자
문서 작업 후 "저장" 버튼을 누르는 것과 같습니다.
단, "채용공고 수정"처럼 메모를 남기면서 저장합니다.

### 백엔드 개발자
git에서 스테이지된 변경사항을 스냅샷으로 기록하는 것입니다.
`git commit -m "message"`로 변경 이력을 남깁니다.

### 프론트엔드 개발자
코드 변경을 "세이브 포인트"로 기록하는 것입니다.
Ctrl+S가 파일 저장이면, commit은 프로젝트 전체 상태 저장입니다.

### 풀스택 개발자
변경사항의 원자적(atomic) 스냅샷입니다.
의미 있는 단위로 변경을 묶어서 이력에 기록합니다.

### UI/UX 디자이너
Figma에서 "버전 저장"하는 것과 같습니다.
"로그인 화면 수정" 같은 설명을 달면서 현재 상태를 기록합니다.

### 제품 기획자/PM
Google Docs에서 "중요 변경사항 메모"를 남기는 것과 같습니다.
"요구사항 v2 반영"처럼 왜 바꿨는지 기록하면서 저장합니다.

### 데이터 분석가
실험 로그에 "실험 #5: learning rate 0.01" 같은 체크포인트를 남기는 것입니다.
나중에 어떤 시도가 효과적이었는지 추적할 수 있습니다.

### 마케터/콘텐츠 제작자
카피 수정 후 "수정 이유: A/B 테스트 결과 반영"이라고 메모하며 저장하는 것입니다.
나중에 왜 바꿨는지 바로 확인할 수 있습니다.

### 일반 사무/관리
업무일지에 "오늘 한 일" 한 줄 적으면서 저장하는 것입니다.
"연말정산 서류 업데이트"처럼 무엇을 바꿨는지 기록합니다.

### 연구원/학생
실험 노트에 날짜와 변경 내용을 기록하는 것입니다.
"방법론 섹션 재작성"처럼 메모를 남기면서 저장합니다.

## 5. CLAUDE.md

### HR/인사 담당자
신입 사원에게 주는 "업무 매뉴얼"과 같습니다.
"우리 회사는 이렇게 일해"를 적어놓으면 Claude가 매번 읽고 따릅니다.

### 백엔드 개발자
프로젝트 규칙 설정 파일(.eslintrc, .editorconfig)과 같습니다.
코딩 컨벤션, 아키텍처 규칙 등을 Claude에게 영구적으로 전달합니다.

### 프론트엔드 개발자
.prettierrc나 .eslintrc 같은 프로젝트 설정 파일입니다.
"우리 팀은 세미콜론 안 써" 같은 규칙을 Claude가 항상 따릅니다.

### 풀스택 개발자
프로젝트 루트의 설정 파일(Makefile, docker-compose.yml)과 비슷합니다.
프로젝트 전체 규칙과 워크플로우를 Claude에게 전달합니다.

### UI/UX 디자이너
디자인 시스템 가이드라인 문서와 같습니다.
"Primary 컬러는 #FF6B00", "여백은 8px 단위" 같은 규칙을 저장합니다.

### 제품 기획자/PM
프로젝트 킥오프 문서와 같습니다.
"이 프로젝트 목표는 OO, 타겟은 OO"을 적으면 Claude가 항상 참고합니다.

### 데이터 분석가
프로젝트 데이터 사전(Data Dictionary)과 같습니다.
"user_id는 고유번호, created_at은 UTC" 같은 정의를 저장합니다.

### 마케터/콘텐츠 제작자
브랜드 톤앤매너 가이드와 같습니다.
"우리는 친근한 반말체, 이모지 사용 금지" 같은 규칙을 저장합니다.

### 일반 사무/관리
팀 업무 규칙서와 같습니다.
"보고서는 한글, 제목은 OO 형식" 같은 규칙을 저장하면 Claude가 따릅니다.

### 연구원/학생
연구실 프로토콜 문서와 같습니다.
"APA 형식 사용, 수동태 금지" 같은 논문 작성 규칙을 저장합니다.

npm 스크립트나 VS Code 스니펫과 같습니다.

### 풀스택 개발자
Makefile이나 GitHub Actions 워크플로우와 비슷합니다.

### UI/UX 디자이너
Figma 플러그인과 같습니다.
"디자인 QA 체크리스트" 같은 반복 작업을 자동화합니다.

### 제품 기획자/PM
Notion 템플릿과 같습니다.
"스프린트 회고 문서 작성" 같은 반복 작업을 한 번 만들면 자동으로 실행됩니다.

### 데이터 분석가
Jupyter Notebook 템플릿이나 재사용 함수 라이브러리와 같습니다.
"데이터 EDA 자동화" 같은 분석 워크플로우를 스킬로 만들 수 있습니다.

### 마케터/콘텐츠 제작자
콘텐츠 템플릿과 같습니다.
"블로그 글 작성" 같은 반복 작업을 스킬로 만들면 형식/톤을 자동으로 맞춰줍니다.

### 일반 사무/관리
Excel 매크로와 같습니다.
"월간 보고서 만들기" 같은 반복 작업을 한 번 설정하면 매번 자동으로 처리됩니다.

### 연구원/학생
실험 프로토콜 자동화와 같습니다.
"논문 초록 작성" 같은 반복 작업을 스킬로 만들면 형식을 자동으로 맞춰줍니다.

## 7. MCP (Model Context Protocol)

### HR/인사 담당자
채용 사이트(잡코리아, 사람인)와 자동 연동하는 것과 같습니다.
Claude가 외부 서비스의 데이터를 직접 가져와서 활용합니다.

### 백엔드 개발자
REST API 클라이언트를 Claude에 플러그인하는 프로토콜입니다.
외부 서비스(DB, API 등)에 Claude가 직접 접근할 수 있게 합니다.

### 프론트엔드 개발자
API 연동 미들웨어와 같습니다.
Claude가 Figma, GitHub 등 외부 서비스를 직접 호출할 수 있게 합니다.

### 풀스택 개발자
마이크로서비스 간 통신 프로토콜과 비슷합니다.
Claude를 외부 시스템(DB, 클라우드, API)에 연결하는 표준 인터페이스입니다.

### UI/UX 디자이너
Figma와 다른 도구(Zeplin, Jira)를 자동 연동하는 것과 같습니다.
Claude가 디자인 도구의 데이터를 직접 읽고 활용할 수 있게 합니다.

### 제품 기획자/PM
Jira, Slack, Notion을 하나로 연결하는 Zapier/Make와 같습니다.
Claude가 여러 서비스의 데이터를 자동으로 가져와서 종합합니다.

### 데이터 분석가
데이터베이스 커넥터(ODBC/JDBC)와 같습니다.
Claude가 DB, API, 파일 시스템의 데이터를 직접 조회하고 분석합니다.

### 마케터/콘텐츠 제작자
마케팅 도구(GA, Meta Ads)를 자동 연동하는 것과 같습니다.
Claude가 광고 성과 데이터를 직접 가져와서 분석해줍니다.

### 일반 사무/관리
업무 도구(이메일, 캘린더, 파일)를 한 곳에서 연결하는 것과 같습니다.
Claude가 여러 프로그램의 정보를 자동으로 가져옵니다.

### 연구원/학생
논문 DB(PubMed, arXiv)와 자동 연동하는 것과 같습니다.
Claude가 최신 논문을 직접 검색하고 요약해줍니다.

## 8. Subagent

### HR/인사 담당자
업무를 분담하는 "팀원"과 같습니다.
Claude가 혼자 하기 어려운 일을 작은 전문가들에게 나눠서 시킵니다.

### 백엔드 개발자
마이크로서비스 패턴의 워커(worker)입니다.
메인 에이전트가 복잡한 작업을 서브태스크로 분리해 병렬 실행합니다.

### 프론트엔드 개발자

## 10. Hook

### HR/인사 담당자

### 백엔드 개발자

pre-commit hook으로 린트 체크하는 것과 같은 원리입니다.

### 풀스택 개발자
CI/CD 파이프라인의 트리거와 같습니다.
특정 이벤트(push, PR 등) 발생 시 자동으로 정의된 작업을 실행합니다.

### UI/UX 디자이너
Figma에서 "컴포넌트 변경 시 모든 인스턴스 자동 업데이트"와 같습니다.
특정 상황이 발생하면 자동으로 지정한 작업이 실행됩니다.

### 제품 기획자/PM

### 데이터 분석가

"새 데이터 들어오면 자동으로 전처리 실행" 같은 자동화 규칙입니다.

### 마케터/콘텐츠 제작자
이메일 자동화(트리거 메일)와 같습니다.
"이 조건이 되면 자동으로 이 작업 실행" 같은 규칙을 설정합니다.

### 일반 사무/관리
Outlook 규칙("특정 메일 받으면 자동 분류")과 같습니다.
"이 상황이 되면 자동으로 이것을 해줘"라는 자동화 규칙입니다.

### 연구원/학생

원본 디자인은 그대로 두고, 새 버전을 따로 만들어서 작업합니다.

### 제품 기획자/PM
기획서의 "대안 시나리오"를 따로 작성하는 것과 같습니다.
원본 기획서는 유지하면서, 새로운 방향을 별도로 검토합니다.

### 데이터 분석가
분석의 "실험 환경"과 같습니다.
메인 분석은 그대로 두고, 새로운 가설을 별도 환경에서 테스트합니다.

### 마케터/콘텐츠 제작자
캠페인의 "B안"을 따로 만드는 것과 같습니다.
A안(원본)은 유지하면서, 새로운 시도를 별도로 진행합니다.

### 일반 사무/관리
보고서의 "사본"을 만들어서 수정하는 것과 같습니다.
원본 보고서는 건드리지 않고, 사본에서 마음껏 고친 뒤 합칩니다.

### 연구원/학생
논문의 "수정 초안"을 따로 작성하는 것과 같습니다.
현재 버전은 보존하면서, 새로운 접근법을 별도로 시도합니다.
